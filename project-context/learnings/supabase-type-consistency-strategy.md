# Supabase Type Consistency Strategy
# Backend-Mobile Integration Type Safety

**Date**: 2025-10-21
**Status**: 🎯 Production-Ready Strategy
**Context**: Prevent type mismatches between Supabase backend and React Native mobile app

---

## 📋 Executive Summary

**Problem**: Backend type changes (functions, tables, views) can silently break mobile app without detection
**Root Cause Discovered**: Backend `get_organisation_users` function changed but mobile app types were never regenerated
**Impact**: 2+ days debugging ERROR 42883 function signature mismatches
**Solution**: Automated type generation + validation + CI/CD integration

---

## 🎯 Core Strategy: 3-Layer Type Safety

### Layer 1: Type Generation (Automated)
Generate TypeScript types from **cloud production database** (NOT local)

### Layer 2: Type Validation (Pre-Deployment)
Validate function signatures match before deploying mobile app

### Layer 3: Runtime Verification (Optional)
Validate API responses match expected types at runtime

---

## 🔧 Implementation Guide

### 1. Supabase Type Generation

#### **Command Reference** (Supabase CLI v1.8.1+)

```bash
# Generate from CLOUD production database (RECOMMENDED)
npx supabase gen types typescript \
  --project-id "$PROJECT_REF" \
  --schema public,storage \
  > src/types/supabase.ts

# Generate from LOCAL database (development only)
npx supabase gen types typescript --local > src/types/supabase-local.ts

# Generate from LINKED project (after `supabase link`)
npx supabase gen types typescript --linked > src/types/supabase.ts
```

#### **Why Cloud Production?**
- ✅ **Source of Truth**: Cloud production has the deployed schema
- ✅ **No Drift**: Local databases can have uncommitted changes
- ✅ **Integration Testing**: Mobile app tests against cloud dev/staging
- ❌ **Avoid Local**: Local database might have experimental changes

#### **Project Reference ID**
```bash
# Get from Supabase Dashboard URL
https://supabase.com/dashboard/project/<PROJECT_REF>

# Or from local config
cat .env | grep SUPABASE_URL
# Extract from: https://<PROJECT_REF>.supabase.co
```

### 5. Function Signature Validation

#### **Automated Function Contract Testing**

Create `scripts/validate-function-signatures.ts`:

```typescript
/**
 * Validates that RPC function signatures match expected types
 * Run before deployment to catch backend contract changes
 */

import { createClient } from '@supabase/supabase-js';
import { Database } from '../src/types/supabase';

const supabase = createClient<Database>(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!
);

interface FunctionSignature {
  name: string;
  args: Record<string, string>; // parameter_name -> expected_type
  returns: string;
}

const EXPECTED_SIGNATURES: FunctionSignature[] = [
  {
    name: 'has_system_role',
    args: {
      user_id: 'uuid',
      required_role: 'text'
    },
    returns: 'boolean'
  },
  {
    name: 'has_project_role',
    args: {
      user_id: 'uuid',
      project_id: 'uuid',
      required_role: 'text'
    },
    returns: 'boolean'
  },
  {
    name: 'get_organisation_users',
    args: {
      p_organisation_id: 'uuid',
      p_user_id: 'uuid'
    },
    returns: 'TABLE(user_id uuid, name text, email text, roles jsonb)'
  }
];

async function validateFunctionSignatures() {
  console.log('🔍 Validating function signatures against production...\n');

  let hasErrors = false;

  for (const expected of EXPECTED_SIGNATURES) {
    // Query pg_catalog to get actual function signature
    const { data, error } = await supabase.rpc('execute_sql', {
      query: `
        SELECT
          p.proname as name,
          pg_get_function_arguments(p.oid) as args,
          pg_get_function_result(p.oid) as returns
        FROM pg_proc p
        JOIN pg_namespace n ON p.pronamespace = n.oid
        WHERE n.nspname = 'public'
          AND p.proname = '${expected.name}'
      `
    });

    if (error) {
      console.error(`❌ ${expected.name}: Failed to query function`);
      console.error(error.message);
      hasErrors = true;
      continue;
    }

    if (!data || data.length === 0) {
      console.error(`❌ ${expected.name}: Function not found in database`);
      hasErrors = true;
      continue;
    }

    const actual = data[0];

    // Parse and compare arguments
    const actualArgs = parseArguments(actual.args);
    const expectedArgs = Object.entries(expected.args);

    let argsMatch = true;
    for (const [paramName, paramType] of expectedArgs) {
      if (!actualArgs[paramName] || actualArgs[paramName] !== paramType) {
        console.error(
          `❌ ${expected.name}: Parameter mismatch\n` +
          `   Expected: ${paramName} ${paramType}\n` +
          `   Actual:   ${paramName} ${actualArgs[paramName] || 'MISSING'}`
        );
        argsMatch = false;
        hasErrors = true;
      }
    }

    // Compare return type
    if (actual.returns !== expected.returns) {
      console.error(
        `❌ ${expected.name}: Return type mismatch\n` +
        `   Expected: ${expected.returns}\n` +
        `   Actual:   ${actual.returns}`
      );
      hasErrors = true;
    }

    if (argsMatch && actual.returns === expected.returns) {
      console.log(`✅ ${expected.name}: Signature valid`);
    }
  }

  if (hasErrors) {
    console.error('\n❌ Function signature validation FAILED');
    process.exit(1);
  }

  console.log('\n✅ All function signatures valid');
}

function parseArguments(argsString: string): Record<string, string> {
  // Parse "user_id uuid, required_role text" -> { user_id: 'uuid', required_role: 'text' }
  const result: Record<string, string> = {};

  if (!argsString) return result;

  const params = argsString.split(',').map(p => p.trim());
  for (const param of params) {
    const [name, type] = param.split(' ');
    if (name && type) {
      result[name] = type;
    }
  }

  return result;
}

validateFunctionSignatures().catch(console.error);
```

### 6. Table Schema Alignment

#### **Detect Table Schema Drift**

Create `scripts/validate-table-schemas.ts`:

```typescript
/**
 * Validates that critical table schemas match expected structure
 * Detects field renames, type changes, missing columns
 */

import { createClient } from '@supabase/supabase-js';
import { Database } from '../src/types/supabase';

const supabase = createClient<Database>(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!
);

interface TableSchema {
  table: string;
  columns: {
    name: string;
    type: string;
    nullable: boolean;
  }[];
}

const EXPECTED_SCHEMAS: TableSchema[] = [
  {
    table: 'projects',
    columns: [
      { name: 'id', type: 'uuid', nullable: false },
      { name: 'name', type: 'text', nullable: false },
      { name: 'description', type: 'text', nullable: true },
      { name: 'organisation_id', type: 'uuid', nullable: false },
      { name: 'created_at', type: 'timestamp with time zone', nullable: false },
      { name: 'updated_at', type: 'timestamp with time zone', nullable: false }
    ]
  },
  {
    table: 'project_members',
    columns: [
      { name: 'id', type: 'uuid', nullable: false },
      { name: 'project_id', type: 'uuid', nullable: false },
      { name: 'user_id', type: 'uuid', nullable: false },
      { name: 'role', type: 'text', nullable: false },
      { name: 'created_at', type: 'timestamp with time zone', nullable: false }
    ]
  }
];

async function validateTableSchemas() {
  console.log('🔍 Validating table schemas against production...\n');

  let hasErrors = false;

  for (const expected of EXPECTED_SCHEMAS) {
    // Query information_schema for actual table structure
    const { data, error } = await supabase
      .rpc('execute_sql', {
        query: `
          SELECT
            column_name,
            data_type,
            is_nullable
          FROM information_schema.columns
          WHERE table_schema = 'public'
            AND table_name = '${expected.table}'
          ORDER BY ordinal_position
        `
      });

    if (error) {
      console.error(`❌ ${expected.table}: Failed to query schema`);
      console.error(error.message);
      hasErrors = true;
      continue;
    }

    if (!data || data.length === 0) {
      console.error(`❌ ${expected.table}: Table not found`);
      hasErrors = true;
      continue;
    }

    // Build actual schema map
    const actualColumns = new Map(
      data.map(col => [
        col.column_name,
        {
          type: col.data_type,
          nullable: col.is_nullable === 'YES'
        }
      ])
    );

    // Validate each expected column
    let tableValid = true;
    for (const expectedCol of expected.columns) {
      const actualCol = actualColumns.get(expectedCol.name);

      if (!actualCol) {
        console.error(
          `❌ ${expected.table}.${expectedCol.name}: Column missing in database`
        );
        tableValid = false;
        hasErrors = true;
        continue;
      }

      if (actualCol.type !== expectedCol.type) {
        console.error(
          `❌ ${expected.table}.${expectedCol.name}: Type mismatch\n` +
          `   Expected: ${expectedCol.type}\n` +
          `   Actual:   ${actualCol.type}`
        );
        tableValid = false;
        hasErrors = true;
      }

      if (actualCol.nullable !== expectedCol.nullable) {
        console.error(
          `❌ ${expected.table}.${expectedCol.name}: Nullability mismatch\n` +
          `   Expected: ${expectedCol.nullable ? 'nullable' : 'not null'}\n` +
          `   Actual:   ${actualCol.nullable ? 'nullable' : 'not null'}`
        );
        tableValid = false;
        hasErrors = true;
      }
    }

    if (tableValid) {
      console.log(`✅ ${expected.table}: Schema valid (${expected.columns.length} columns)`);
    }
  }

  if (hasErrors) {
    console.error('\n❌ Table schema validation FAILED');
    process.exit(1);
  }

  console.log('\n✅ All table schemas valid');
}

validateTableSchemas().catch(console.error);
```

Create `src/types/api-schemas.ts`:

```typescript
import { z } from 'zod';

/**
 * Zod schemas for validating API responses at runtime
 * Catches type mismatches that TypeScript can't detect
 */

// has_system_role response schema
export const HasSystemRoleSchema = z.boolean();

// get_organisation_users response schema
export const OrganisationUserSchema = z.object({
  user_id: z.string().uuid(),
  name: z.string(),
  email: z.string().email(),
  roles: z.array(z.object({
    role: z.string(),
    organisation_id: z.string().uuid().optional(),
    project_id: z.string().uuid().optional()
  }))
});

export const OrganisationUsersSchema = z.array(OrganisationUserSchema);

// Project member schema
export const ProjectMemberSchema = z.object({
  id: z.string().uuid(),
  user_id: z.string().uuid(),
  project_id: z.string().uuid(),
  role: z.string(),
  user_profile: z.object({
    full_name: z.string().nullable(),
    email: z.string().email()
  }).optional()
});

export const ProjectMembersSchema = z.array(ProjectMemberSchema);
```

Use in service layer:

```typescript
import { OrganisationUsersSchema } from '@/types/api-schemas';

async function getOrganisationUsers(orgId: string, userId: string) {
  const { data, error } = await supabase.rpc('get_organisation_users', {
    p_organisation_id: orgId,
    p_user_id: userId
  });

  if (error) throw error;

  // Validate runtime response matches expected schema
  try {
    const validatedData = OrganisationUsersSchema.parse(data);
    return validatedData;
  } catch (zodError) {
    // Type mismatch detected at runtime!
    console.error('❌ API response type mismatch:', zodError);
    throw new Error(`API contract violation: ${zodError.message}`);
  }
}
```

## 📊 Type Drift Detection Strategy

### What Causes Type Drift?

| Change Type | Example | Detection Method |
|-------------|---------|------------------|
| **Function Parameter Renamed** | `user_id` → `check_user_id` | Function signature validation |
| **Function Parameter Type Changed** | `text` → `varchar(255)` | Function signature validation |
| **New Required Column Added** | `projects.status` | Table schema validation |
| **Column Type Changed** | `id: number` → `id: uuid` | TypeScript compilation + table schema validation |
| **Function Return Type Changed** | `boolean` → `jsonb` | Function signature validation |
| **Table Renamed** | `users` → `user_profiles` | TypeScript compilation (SELECT * fails) |

### Detection Timeline

```
Backend Deployment → Type Drift Occurs
     ↓
Nightly Type Sync (2 AM UTC) → Detects drift within 24 hours
     ↓
Auto-PR Created → Developer notified
     ↓
Developer Reviews → Merges or fixes breaking changes
     ↓
Pre-Deployment Validation → Blocks deploy if types stale
```

## 🚨 Breaking Change Protocol

### When Backend Makes Breaking Changes

**Backend Team Must**:
1. Create migration with backward-compatible transition period (if possible)
2. Notify mobile team via cross-repo webhook OR Slack
3. Document breaking change in migration file
4. Wait for mobile team confirmation before removing old API

**Mobile Team Must**:
1. Review auto-generated type PR immediately
2. Update service layer to handle new signatures
3. Test integration against cloud dev database
4. Deploy mobile app update
5. Confirm to backend team that migration complete

**Example: Renaming Function Parameter**

Backend migration:
```sql
-- Step 1: Add new parameter, keep old one (backward compatible)
CREATE OR REPLACE FUNCTION has_system_role(
  user_id uuid,           -- OLD (deprecated)
  check_user_id uuid,     -- NEW
  required_role text
) RETURNS boolean AS $$
BEGIN
  -- Support both old and new parameter names
  IF check_user_id IS NULL THEN
    check_user_id := user_id;
  END IF;

  -- Function logic using check_user_id
END;
$$ LANGUAGE plpgsql;

-- Step 2: After mobile app updated, remove old parameter in next migration
```

## 📚 Best Practices

### ✅ DO

- **Generate from Cloud Production**: Always use `--project-id` flag, not `--local`
- **Commit Generated Types**: Check `supabase.ts` into Git for version control
- **Automate Regeneration**: Use GitHub Actions for nightly sync
- **Block Stale Deploys**: Fail CI if types out of sync
- **Document Breaking Changes**: Backend team must notify mobile team
- **Test After Type Changes**: Run integration tests against cloud dev
- **Use Strict TypeScript**: Enable `strict: true` in `tsconfig.json`

### ❌ DON'T

- **Don't Edit Generated Types**: Never manually modify `supabase.ts`
- **Don't Generate from Local**: Local database can have uncommitted changes
- **Don't Skip Type Regeneration**: After every backend schema change, regenerate
- **Don't Deploy Without Validation**: Always run `npm run types:check` pre-deploy
- **Don't Ignore Type Warnings**: TypeScript errors indicate contract mismatches
- **Don't Mix Type Sources**: Pick one strategy (cloud, local, OR linked)

## 🔍 Troubleshooting

### Issue: Types not regenerating

**Symptoms**: `npm run types:generate` doesn't update file

**Solutions**:
1. Verify `SUPABASE_ACCESS_TOKEN` is valid: `supabase login`
2. Check `PROJECT_REF` is correct: Compare with dashboard URL
3. Verify schema name: Use `public,storage` not `public, storage` (no spaces)
4. Test CLI directly: `npx supabase gen types typescript --project-id "$PROJECT_REF"`

### Issue: GitHub Action fails with "401 Unauthorized"

**Cause**: Invalid or missing `SUPABASE_ACCESS_TOKEN`

**Solution**:
1. Generate new token: https://supabase.com/dashboard/account/tokens
2. Update GitHub Secret: Settings → Secrets → `SUPABASE_ACCESS_TOKEN`
3. Re-run workflow

### Issue: Type mismatch errors after regeneration

**Cause**: Backend schema changed, mobile code not updated

**Solution**:
1. Review `git diff src/types/supabase.ts`
2. Update service layer code to match new types
3. Search codebase for affected RPC calls: `grep -r "supabase.rpc('function_name'"`
4. Run tests: `npm test`

### Issue: Function signature validation fails

**Cause**: Backend function parameters or return type changed

**Solution**:
1. Check backend migration files for function changes
2. Update `EXPECTED_SIGNATURES` in `scripts/validate-function-signatures.ts`
3. Update mobile service calls to match new signature
4. Re-run validation: `npm run validate:functions`

## 🎓 Learning from the get_organisation_users Incident

### What Went Wrong

**Timeline**:
1. Backend updated `get_organisation_users` function (added schema qualification)
2. Mobile types never regenerated
3. Mobile app called with outdated type assumptions
4. PostgreSQL threw ERROR 42883 (function signature mismatch)
5. 2+ days debugging

### Root Causes

1. **No Automated Type Sync**: Types were manually managed
2. **No Pre-Deployment Validation**: Deploy didn't check type freshness
3. **No Cross-Repo Communication**: Backend changes invisible to mobile team
4. **Local vs Cloud Mismatch**: Types generated from local, tested against cloud

### Prevention Measures Implemented

1. ✅ **Nightly Type Sync**: Auto-detects backend changes within 24 hours
2. ✅ **Pre-Deployment Validation**: Blocks deploy if types stale
3. ✅ **Function Signature Validation**: Catches parameter mismatches
4. ✅ **Cross-Repo Webhook** (optional): Immediate notification on backend deploy
5. ✅ **Cloud Production Source**: Types always match deployed backend
