**Current State**: Default journal mode
**Target State**: WAL mode enabled
**Impact**: 5-10x write performance, sync doesn't block UI

**Implementation**:
- [x] Add `PRAGMA journal_mode = WAL;` to DatabaseService initialization
- [ ] Test offline sync performance
- [ ] Verify concurrent read/write operations
- [ ] Document in code comments
- [ ] Update testing guide with WAL mode

**Code Change** (src/services/offline/DatabaseService.ts):
```typescript
async initializeDatabase(): Promise<void> {
  this.db = await SQLite.openDatabaseAsync(this.DATABASE_NAME);

  await this.db.execAsync('PRAGMA foreign_keys = ON;');
  await this.db.execAsync('PRAGMA journal_mode = WAL;');  // 👈 ADD THIS

  // ... rest of initialization
}
```

**Testing**:
```typescript

expect(result.journal_mode).toBe('wal');
```

### 24.6: Production Security Audit (MEDIUM PRIORITY - 1 hour)

**Why**: Console.log in production exposes sensitive data. Environment variables may be misconfigured.

**Security Checklist**:
- [ ] Search for `console.log` in src/ directory
- [ ] Remove or wrap in `__DEV__` checks
- [ ] Configure babel to remove console in production
- [ ] Audit .env.local for `EXPO_PUBLIC_` prefix
- [ ] Verify SecureStore usage for auth tokens
- [ ] Check for hardcoded secrets
- [ ] Validate HTTPS-only configuration
- [ ] Review Supabase RLS policies

**Implementation**:

**1. Console.log Removal** (babel.config.js):
```javascript
module.exports = {
  plugins: [
    process.env.NODE_ENV === 'production' &&
      'transform-remove-console'
  ].filter(Boolean)
};
```

**2. Environment Variable Audit**:
```bash
# Check for sensitive keys with wrong prefix
grep -E "PASSWORD|SECRET|PRIVATE" .env.local
```

**3. SecureStore Verification**:
```typescript
// ✅ GOOD: Secure storage
import * as SecureStore from 'expo-secure-store';
await SecureStore.setItemAsync('auth_token', token);

// ❌ BAD: Insecure storage
import AsyncStorage from '@react-native-async-storage/async-storage';
await AsyncStorage.setItem('auth_token', token);  // FIX THIS
```

**Acceptance Criteria**:
- Zero console.log statements in production build
- All environment variables correctly prefixed
- Auth tokens stored in SecureStore only
- No hardcoded secrets found
- Security checklist 8/10 items complete

**Configuration**:
```typescript

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  environment: process.env.NODE_ENV,
```

### 24.8: RLS Performance Optimization (LOW PRIORITY - 2 hours)

**Why**: Proper indexing and function wrapping provides 2-3x query performance improvement on complex policies

**Current State**: RLS policies exist but may be unoptimized
**Target State**: Indexed and optimized RLS policies
**Impact**: 2-3x query performance

**Implementation**:
- [ ] Audit existing RLS policies in Supabase
- [ ] Add indexes on `user_id` columns
- [ ] Wrap `auth.uid()` in functions (caching)
- [ ] Benchmark query performance (before/after)
- [ ] Document optimization patterns
- [ ] Update backend coordination docs

**Optimization Pattern**:
```sql
-- ✅ GOOD: Optimized RLS with index
CREATE POLICY "Users can view own deployments"
ON deployments FOR SELECT
TO authenticated
USING (user_id = (SELECT auth.uid()));  -- Cached per statement

CREATE INDEX idx_deployments_user_id ON deployments(user_id);

-- ❌ BAD: Unoptimized RLS
USING (user_id = auth.uid());  -- Called for every row
```

**Acceptance Criteria**:
- All RLS policies audited
- Critical tables have user_id indexes
- `auth.uid()` wrapped in SELECT statements
- Performance improvement measured
- Patterns documented for future tables

**Note**: Requires backend coordination - create task in backend repo
