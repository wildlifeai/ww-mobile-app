PARALLEL DEVELOPMENT STREAM A - Task 12: Projects CRUD Operations

This task implements the complete project management system for Wildlife Watcher with organisation multi-tenancy, WW Admin global oversight, and LoRaWAN webhook integration per Implementation Spec v1.4.6.

## Implementation Requirements:

### **ENHANCED** 12.1: Projects List Screen with Organisation Context
- Create `src/navigation/screens/ProjectsScreen.tsx` following Component Implementation Patterns
- **NEW**: Maestro TDD implementation for project screen components
- Project cards with summary information and role-based visibility
- **NEW**: Organisation filtering for multi-org users with dropdown selection
- **NEW**: WW Admin view (all projects across all organisations) - read-only visibility only
- **NEW**: Role-based project access:
  - `ww_admin`: View all projects across all organisations
  - `project_admin`: View organisation projects they administer 
  - `project_member`: View projects they're members of
- Pull-to-refresh functionality with offline support
- Search and filter capabilities with organisation scoping
- **NEW**: LoRaWAN device status indicators on project cards
- Navigation to project details with permission validation
- **NEW**: Claude Flow SPARC navigation patterns

### 12.2: New Project Creation
- Create `src/navigation/screens/NewProjectScreen.tsx`
- Multi-section form with react-hook-form:
  - **Basic Information**: name, description, sampling design
  - **Project Settings**: private/public, using bait, monitoring marked
  - **Team Members**: add by email with role assignment
- Comprehensive form validation and error handling
- Integration with Supabase projects table
- Offline support with sync queue integration
- Success/error feedback with navigation

### 12.3: Project Service Layer with Organisation Multi-Tenancy
- Create `src/services/ProjectService.ts` extending BaseService with organisation context
- **NEW**: Maestro test coverage for all service operations
- Implement full CRUD operations with role-based access:
  - `createProject()` - with offline queue support and organisation scoping
  - `getUserProjects()` - role-filtered with computed fields (member count, deployment counts, LoRaWAN device counts)
  - `updateProject()` - with optimistic updates and permission validation
  - `deleteProject()` - with confirmation, cleanup, and admin-only access
- **NEW**: Organisation-aware member management operations:
  - `addProjectMember()` with role assignment (project_admin, project_member)
  - `removeProjectMember()` with permission validation
  - `getProjectMembers()` with profile information and role display
- **NEW**: WW Admin operations (read-only only):
  - `getAllProjectsAdmin()` - cross-organisation project access (read-only)
- **NEW**: LoRaWAN integration operations:
  - `getProjectDeviceStatus()` - device battery and storage status
  - `syncProjectDevices()` - webhook integration support
- Type-safe operations using generated Supabase types with organisation and role extensions

### 12.4: RTK Query Integration
- Create `src/store/api/projectsApi.ts`
- Implement optimized queries with proper caching:
  - `getProjects` with tag-based invalidation
  - `createProject` with optimistic updates
  - `updateProject` with immediate UI feedback
  - `deleteProject` with proper cleanup
- Handle loading states and error scenarios
- Integration with offline service for queue management
