# Dev Database Reset Guide

> **Related**: [03-DATA-AND-SYNC.md](../onboarding/03-DATA-AND-SYNC.md) covers the WatermelonDB offline data layer and Supabase sync architecture.

## Overview

Development-only utility to reset the local WatermelonDB database when testing with new seed data from the cloud dev instance or local Supabase.

> [!WARNING]
> Only available in development mode (`__DEV__ === true`). Will throw in production builds.

## Use Cases

- **New seed data deployed** — Backend team has deployed fresh test data to cloud dev
- **Testing multi-tenancy** — Need to clear old org/project data and sync fresh
- **Database corruption** — Local WatermelonDB has inconsistent data
- **Schema changes** — After backend migrations, need a clean slate

## How to Use

### Programmatic Reset

```typescript
import { resetDatabaseForDev } from '@/utils/devDatabaseReset';

// Resets WatermelonDB completely (drops all data + schema, recreates from scratch)
await resetDatabaseForDev();

// Alias — identical behaviour
import { clearDatabaseDataForDev } from '@/utils/devDatabaseReset';
await clearDatabaseDataForDev();
```

Both functions call `database.unsafeResetDatabase()` which drops and rebuilds the entire WatermelonDB database.

### Post-Reset Steps

1. **Restart the app** (recommended)
2. **Log out and log back in** — triggers:
   - Fresh authentication
   - Initial sync from Supabase
   - Download of organisation, project, device, and deployment data

## Troubleshooting

| Error | Cause | Solution |
|-------|-------|----------|
| "Database reset is only available in development mode" | Running production build or `__DEV__` is false | Use development build with `npx expo run:android` |
| Database still has old data after reset | Need to re-authenticate to trigger sync | Log out → close app → reopen → log in |

## Technical Details

**Files**:

| File | Purpose |
|------|---------|
| `src/utils/devDatabaseReset.ts` | Reset utility functions (~64 lines) |
| `src/database/schema.ts` | WatermelonDB schema (auto-generated, ~466 lines) |

**Database**: WatermelonDB (schema version 194, auto-generated by `npm run schema:generate`)

**Key tables** (subset — full list in `src/database/schema.ts`):
- `deployments`, `deployment_statuses`
- `devices`, `device_preparation`
- `projects`, `organisations`, `user_organisations`
- `ai_models`, `ai_model_organisation`
- `api_logs`, `activity_sensitivity`, `capture_methods`

The reset calls `database.unsafeResetDatabase()` which drops all tables and recreates them from the schema definition. No manual table ordering is needed — WatermelonDB handles this internally.
